# 多策略共用账户：统一调度与可靠回调修复方案

> 文档状态：设计待实施；本文不修改现有 Gateway、Helper 或 QMT 下单逻辑。
> 编写日期：2026-07-23。
> 适用范围：一台 Windows、一个大 QMT 账户、一个现有本机 Gateway、多个外置策略进程。
> 前置条件：外置策略不能使用 `xtquant`，所有真实交易仍必须经大 QMT 内置 Python Helper 执行。

## 1. 结论先行

当前 Gateway 的“每账户一个 primary 可靠事件消费者”是正确的可靠性边界，不能通过允许多个策略直接连接 Gateway 来解决多策略问题。若多个策略进程各自调用 `qmt_local_api`：后完成握手的连接会替换旧 primary，可靠订单/成交回调将被切换到不同进程，既无法保证处理顺序，也无法集中风险和关联订单。

正确的修复方式是在 Gateway 外增加一个**账户协调服务**（下文简称 Coordinator）：

```text
策略 A ─┐
策略 B ─┼─ 内部命令 / 事件 API ─→ Account Coordinator
策略 C ─┘                              │
                                       │ 唯一 qmt_local_api 连接
                                       ▼
                              现有 Gateway（保持单 primary）
                                       │
                                       ▼
                            文件队列 Helper → 大 QMT 账户
```

Coordinator 是该账户唯一允许连接 Gateway 的外置进程。它负责：

- 在一个地方执行账户级风险控制、额度预占和策略优先级仲裁；
- 为每个策略生成不冲突且可追溯的 `client_order_id`、`request_id` 与关联记录；
- 先将可靠回调按 `delivery_id` 持久化，再向 Gateway 发送 `DELIVERY_ACK`；
- 将已持久化的账户事件按归属和订阅规则分发给多个策略；
- 在重启后恢复未完成命令、未下发事件与账户对账。

这是一项**外置业务层修复**，不应把 Gateway 改成多 primary、多写者或原始回调广播服务器。

## 2. 问题与根因

### 2.1 当前单账户合同

现有协议明确规定：同一账户只允许一个可靠事件消费者。完成认证的新 TCP 连接将成为 `primary`，旧 primary 被替换。Gateway 对带 `delivery_id` 的事件采用至少一次投递，只有 primary 返回 `DELIVERY_ACK` 后才确认完成。

该合同保护了以下能力：

- 事件有唯一确认方，避免多客户端中“谁 ACK、谁落库”不一致；
- Gateway 可以保存可靠 response/event，并在断线后向唯一会话重投；
- 一个 `client_order_id` 与一个账户幂等记录稳定对应；
- Helper/Gateway 只面对一个顺序明确的外置交易边界。

### 2.2 多策略直连会造成的故障

| 场景 | 直接让多个策略连接 Gateway 的结果 | 后果 |
|---|---|---|
| 策略 B 在策略 A 后完成握手 | B 替换 A 成为 primary | A 不再收到可靠回调，B 却可能收到 A 的订单回调 |
| 两个策略同时对同一股票下单 | Gateway 仅看单笔请求幂等 | 没有账户级净敞口、资金、频率与持仓上限仲裁 |
| 两策略意外使用相同 `client_order_id` | 账户维度发生幂等冲突 | 一方订单被拒绝，或业务归属无法判断 |
| 一个策略 handler 很慢或失败 | 不 ACK、连接断开、Gateway 重投 | 所有账户可靠事件都受该策略影响 |
| 策略崩溃后重启 | 可能被其他策略抢占 primary | 回调恢复路径不确定，人工排障成本高 |

因此，问题不是 TCP 连接数量不足，而是缺少一个“账户级命令仲裁者和事件账本”。

## 3. 目标、边界与非目标

### 3.1 目标

1. 保持一个 QMT 账户仅有一个 Gateway 和一个 Gateway primary。
2. 允许多个策略独立提交意图、订阅其订单状态并读取账户数据。
3. 在真实下单前统一执行账户级风险、额度、仓位与频率限制。
4. 让每一笔订单可以从 QMT 回调反查到来源策略、策略订单和原始意图。
5. 让 Coordinator、策略进程、Gateway 任一重启后都可恢复，不因重复投递造成重复业务动作。
6. 不削弱现有 Gateway/Helper 的身份校验、SQLite 幂等、文件队列 Guard、`SUBMIT_UNKNOWN` 与可靠 ACK 语义。

### 3.2 非目标

- 不把 Gateway 改成多账户、多 primary 或广播型消息总线；
- 不让策略直接写 Helper runtime，或绕过 Gateway 调用 QMT；
- 不承诺将 QMT/券商回调转化为端到端 exactly-once；
- 不把高频全量 Tick 通过 JSON 文件逐笔分发给全部策略；
- 不自动重试 `SUBMIT_UNKNOWN`、`EFFECT_STATE_UNKNOWN` 或其他副作用不确定状态；
- 不在本项目中替换为 `xtquant`、第三方交易接口或跨机器交易架构。

## 4. 目标架构

### 4.1 组件关系

```text
┌────────────────────────────────────────────────────────────┐
│                   外置策略进程（多个）                       │
│  strategy-a     strategy-b      strategy-c                  │
│       │                │               │                    │
└───────┼────────────────┼───────────────┼────────────────────┘
        │                │               │
        └───────── 内部、已认证的策略 API ────────────────────┐
                                                               ▼
┌────────────────────────────────────────────────────────────┐
│ Account Coordinator（每账户一个实例）                        │
│  ├─ Strategy Registry / 权限                                 │
│  ├─ Risk & Reservation / 全局限额                             │
│  ├─ Command Ledger / 订单映射                                 │
│  ├─ Event Inbox / Outbox / 策略订阅                           │
│  └─ 唯一 LocalQmtApi 实例 + 唯一可靠 ACK                     │
└─────────────────────────┬──────────────────────────────────┘
                          │ TCP v2，唯一 primary
                          ▼
┌────────────────────────────────────────────────────────────┐
│ 现有 Gateway：身份、幂等、SQLite、文件入队、可靠重投          │
└─────────────────────────┬──────────────────────────────────┘
                          ▼
          大 QMT 内置 Helper → QMT / 券商 / 交易账户
```

Coordinator 对策略暴露的是**新的内部 API**，不是把 Gateway TCP 端口转给每个策略。内部 API 可选本机 loopback TCP、Windows Named Pipe 或受控的本机 HTTP/gRPC；协议实现可以替换，但必须满足本文件的命令、落库、认证和幂等约束。单机初始实现推荐使用仅绑定 `127.0.0.1` 的版本化内部 TCP/HTTP 服务，并对每个策略使用独立本机凭据。

### 4.2 必须保持的连接拓扑

| 链路 | 允许数量 | 约束 |
|---|---:|---|
| Coordinator → Gateway | 每账户 1 条活动连接 | 唯一 `qmt_local_api`，唯一 Gateway primary |
| 策略 → Coordinator | 多条 | 不得继承或读取 Gateway 令牌与根 `.env` |
| Gateway → Helper | 每账户 1 个 writer | 继续使用现有原子文件队列和 writer lease |
| Helper → QMT | 由 QMT 单线程控制 | 不允许策略或 Coordinator 绕过 Helper 直接调用 |

策略部署包中不应放入根 `.env`、Gateway 认证令牌或 `qmt_local_api` 的生产连接配置。策略只获得其 Coordinator 身份、可下单范围和内部 API 地址。

## 5. 命令链与订单归属

### 5.1 提交路径

```text
策略提交 OrderIntent(strategy_id, strategy_order_id, intent)
  → Coordinator 验证策略身份、字段和幂等键
  → 原子事务：写 command_ledger + 风险预占
  → 判断账户健康、交易时段、余额/仓位/频率/白名单
  → 生成 Gateway client_order_id、request_id、trace_id
  → 唯一 LocalQmtApi.send_order_async()
  → 保存 Gateway 即时结果
  → 返回“已接受到 Coordinator / 已排队 / 已拒绝”，不伪造 QMT 成功
```

策略只提交“订单意图”，不得自行构造 Gateway `client_order_id`、`request_id` 或 `qmt_user_order_id`。Coordinator 是这些字段的唯一发行者；策略可携带稳定的 `strategy_order_id`，作为它自己的业务重试键。

### 5.2 标识符规范

| 字段 | 责任方 | 建议规则 | 重试规则 |
|---|---|---|---|
| `strategy_id` | 配置中心 | 固定、ASCII、全局唯一，例如 `alpha` | 不变 |
| `strategy_order_id` | 策略 | 策略域内稳定业务 ID | 同一业务订单不变 |
| `client_order_id` | Coordinator | `acct1-alpha-20260723-000001` 一类的账户全局唯一 ASCII ID | 同一订单不变 |
| `request_id` | Coordinator | 下单动作使用稳定 ID；撤单动作使用独立稳定 ID | 网络重试不变 |
| `trace_id` | Coordinator | 关联策略、命令和日志的追踪 ID | 不替代幂等键 |
| `qmt_user_order_id` | Gateway 或 Coordinator | 默认留空，让 Gateway 按现有规则派生；若显式传入，最长 23 字符 | 不得在重试时改变 |

Coordinator 应对 `(account_id, strategy_id, strategy_order_id, action_kind)` 建立唯一约束；再对生成的 `(account_id, client_order_id)` 和 `(account_id, request_id)` 建立唯一约束。不得把时间戳、随机备注或 TCP `msg_id` 当作幂等键。

### 5.3 下单命令状态机

Coordinator 的账本状态与 Gateway 的副作用状态机互补，不能覆盖或篡改 Gateway 状态：

```text
RECEIVED
  → RISK_RESERVED
  → GATEWAY_SUBMITTING
  → BRIDGE_QUEUED / REJECTED / UNKNOWN
  → QMT_SUBMITTED / ORDER_ACTIVE / FILLED / CANCELLED / FAILED
```

含义如下：

| 状态 | 定义 | Coordinator 行为 |
|---|---|---|
| `RECEIVED` | 已收到并校验策略命令 | 尚未允许策略认为交易已开始 |
| `RISK_RESERVED` | 已原子预占资金、数量、仓位或频率额度 | 可在前置拒绝时释放 |
| `GATEWAY_SUBMITTING` | 正在调用唯一 Gateway 客户端 | 崩溃恢复时先查账，不重发 |
| `BRIDGE_QUEUED` | Gateway 已可靠排队 | 等待可靠提交结果与 QMT 回调 |
| `REJECTED` | Gateway 或风控在副作用开始前拒绝 | 按规则释放风险预占 |
| `UNKNOWN` | 是否已经到达 QMT 无法证明 | 冻结关联预占，查询和人工/规则对账 |
| `QMT_SUBMITTED` | Helper/QMT 本地提交结果已知 | 继续等待订单和成交回调 |
| `ORDER_ACTIVE` / `FILLED` / `CANCELLED` | 从 QMT 回调确认的订单生命周期 | 更新策略账本与全局风险占用 |

`ASYNC_ORDER status=SENT, stage=BRIDGE_QUEUED` 不是 QMT 已接收；`ASYNC_CANCEL_RESPONSE stage=CANCEL_SUBMITTED` 也不是券商最终撤单成功。只有订单、成交和错误回调能够推进最终账户状态。

### 5.4 撤单路径

策略提交撤单时，应提供 Coordinator 返回的策略订单引用，而不是随意传入 QMT `order_id`。Coordinator 必须：

1. 从命令账本找到该策略是否拥有此订单；
2. 验证订单仍可撤、当前状态与撤单权限；
3. 为撤单生成独立、稳定的 `request_id`；
4. 调用 `cancel_order_async` 或 `cancel_order_by_sysid_async`；
5. 将 `ASYNC_CANCEL` 和可靠 `ASYNC_CANCEL_RESPONSE` 写入账本；
6. 在 `ORDER_UPDATE` 证实终态前保留必要的风险与待撤状态。

不能因 TCP 超时、Coordinator 重启或策略重复请求而生成新撤单 ID 自动再次执行。出现 `SUBMIT_UNKNOWN` 时必须使用原订单 ID、系统委托号和订单查询对账。

## 6. 账户级风控与仲裁

### 6.1 风控决策顺序

每条策略命令在发送 Gateway 前必须经过同一个串行或可事务化的账户风控入口：

```text
策略权限
  → Gateway / Helper 健康
  → 交易时段与账户可交易状态
  → 标的白名单、方向、价格、数量与订单类型
  → 单策略限额
  → 账户净仓、可用资金、挂单冻结与总敞口
  → 频率、并发挂单与日内损失限制
  → 原子风险预占 + 命令账本
  → 发送 Gateway
```

只有通过全部检查且“风险预占 + 命令账本”成功提交后，Coordinator 才能向 Gateway 发送请求。若 Gateway 明确返回 `effect_started=false` 的前置拒绝，可按同一命令事务释放预占并让策略退避；进入 `GATEWAY_SUBMITTING`、`BRIDGE_QUEUED` 或未知状态后，不得假设订单不存在。

### 6.2 最小风控项目

| 类别 | 必须能力 |
|---|---|
| 策略权限 | 策略启停、允许标的、买卖方向、最大单笔数量/金额 |
| 策略额度 | 单策略资金、净仓、挂单数、每分钟下单/撤单次数 |
| 账户额度 | 总可用资金、总净仓、单标的净仓、总挂单冻结、日内最大损失 |
| 订单校验 | 代码、数量、价格、价格类型、重复意图、交易时段 |
| 健康闸门 | Gateway PONG、Helper readiness、查询是否降级、账户数据新鲜度 |
| 熔断 | 连续拒绝、未知提交、回调滞后、对账差异、风控数据过期 |

资金、仓位和挂单预占必须包含所有策略已经提交但尚未获得最终回调的订单；不能只使用最近一次 QMT 持仓快照。账户数据过期、查询缓存降级、事件积压或发生对账差异时，Coordinator 应拒绝新交易副作用并保留查询/人工恢复通道。

### 6.3 策略仲裁规则

初始版本应采取确定性、可审计的规则，而不是允许多个进程抢占：

- 同一策略内按其 `strategy_order_id` 串行处理重复命令；
- 跨策略按“风险通过时间 + 配置优先级 + FIFO”排序；
- 对同标的、反方向、超额度或自成交风险命令，默认拒绝或等待人工定义的净额化规则；
- 未明确批准前，不做跨策略自动合单、对冲、撤单替换或订单净额化；
- 所有拒绝、降级、熔断和人工放行都写入不可变审计记录。

先实现“拒绝冲突、完整记录”，再根据实际策略需求设计组合净额化。过早自动净额化会破坏订单来源、税费、成交归因和撤单权属。

## 7. 可靠事件入库与向策略分发

### 7.1 Gateway ACK 的正确位置

Coordinator 收到带 `delivery_id` 的 Gateway 事件后，必须在**同一个本地数据库事务**内完成：

1. 以 `delivery_id` 为唯一键写入不可变 `event_inbox`；
2. 更新关联订单、风险预占和账户投影；
3. 为应接收该事件的策略写入独立 `strategy_event_outbox`；
4. 事务提交成功后，才让 `qmt_local_api` handler 返回，由 API 发送 `DELIVERY_ACK`。

```text
Gateway reliable event
  → Coordinator event_inbox(delivery_id UNIQUE)
  → update order/risk projection + strategy_event_outbox
  → COMMIT
  → return from handler
  → qmt_local_api sends DELIVERY_ACK
  → asynchronous delivery to strategy subscribers
```

Coordinator 不应等待每个策略处理完成后才 ACK Gateway：一个慢策略会阻塞全账户事件，并触发 Gateway 重投。Coordinator 已提交自己的 inbox/outbox 后便已接管可靠性；策略侧的投递、重试和确认由 Coordinator 的 outbox 单独管理。

### 7.2 事件路由规则

| 事件类别 | 处理与可见范围 |
|---|---|
| `ASYNC_ORDER_RESPONSE` / `ASYNC_CANCEL_RESPONSE` | 按 `request_id`、`client_order_id` 路由给命令来源策略，同时进入账户审计 |
| `ORDER_UPDATE` / `TRADE_NOTIFY` | 按 `client_order_id`、`qmt_user_order_id`、`order_id`、`order_sysid` 关联；来源策略收到专属事件，风险与审计收到全量事件 |
| `ORDER_ERROR` | 路由给来源策略、风险服务和审计；无法关联时触发人工/自动对账告警 |
| `ASSET_UPDATE` / `POSITIONS_SNAPSHOT` / `QMT_STATUS` | 更新账户投影；可按只读授权向策略广播摘要 |
| `RECONCILE_REQUIRED` / 未知状态 | 进入风控熔断与对账队列；不能仅广播后继续交易 |

策略不应默认获得其他策略订单的逐笔详情。账户级资产、持仓、风险状态可按最小必要权限提供摘要；完整账本只给风险、运维和审计角色。

### 7.3 策略侧投递语义

Coordinator 到策略的事件同样应至少一次投递：策略使用 `coordinator_event_id` 持久化去重，再确认自己的内部 ACK。若策略仅需要通知，可使用可丢失订阅；但订单、成交、风控状态必须使用持久化 outbox。

当策略离线时，Coordinator 不丢弃其专属订单事件。恢复连接后按该策略的 outbox 顺序补投。策略积压超过配额时，暂停该策略或转人工，不影响 Coordinator 对 Gateway 的 ACK 和其他策略的交易路径。

## 8. Coordinator 最小持久化模型

数据库可使用本机 SQLite WAL（单 Coordinator 进程）或受控外部数据库（需要高可用时）。无论实现介质如何，下列唯一约束与事务边界必须保留。

| 表/投影 | 主键或唯一键 | 目的 |
|---|---|---|
| `strategy_registry` | `strategy_id` | 策略状态、权限、额度、优先级和内部认证材料 |
| `strategy_command` | `(account_id, strategy_id, strategy_order_id, action_kind)` | 策略原始意图、命令生命周期、幂等入口 |
| `order_mapping` | `(account_id, client_order_id)`；`request_id` 唯一 | 策略订单、Gateway/QMT 关联键和订单归属 |
| `risk_reservation` | `reservation_id` | 待提交、挂单、未知状态占用的资金/仓位/频率 |
| `event_inbox` | `delivery_id` 唯一 | Gateway 可靠事件的不可变接收账本 |
| `strategy_event_outbox` | `coordinator_event_id` 唯一 | 各策略待投递、已 ACK、重试状态 |
| `account_projection` | `account_id` | 资产、持仓、挂单、健康、新鲜度和熔断状态 |
| `audit_log` | 单调 ID | 风控决定、人工操作、对账和配置版本 |

`event_inbox` 不应因策略 ACK 后立即删除；至少保留到满足审计、对账和回放策略。`delivery_id`、`client_order_id`、`request_id` 与策略订单映射属于安全状态，不应通过短 TTL 清理来“恢复”旧命令的可执行性。

## 9. 故障恢复与对账

### 9.1 Coordinator 重启

```text
启动 Coordinator
  → 打开命令账本、event_inbox、outbox、风险预占
  → 连接 Gateway 并完成 PING/PONG 身份校验
  → 查询 ACCOUNT_STATUS、ORDER、DEAL、POSITION、ASSET
  → 重建账户投影并比对未完成命令
  → 补投 strategy_event_outbox
  → 仅在健康、数据新鲜且无对账差异时开放新交易
```

Coordinator 重启前已经持久化、但尚未对 Gateway ACK 的事件会由 Gateway 重投；已 ACK 的事件从本地 `event_inbox`/outbox 恢复。两种情况都必须以唯一键去重。

### 9.2 不确定提交

对 `SUBMIT_UNKNOWN`、`EFFECT_STATE_UNKNOWN`、`POST_ENQUEUE_STATE_UNCERTAIN`、未能关联的订单/成交回调以及 QMT 账户健康不一致，采取以下固定动作：

1. 将对应 `strategy_command` 与风险预占置为 `UNKNOWN`；
2. 暂停该策略或该账户的新交易（按影响范围配置）；
3. 使用原 `request_id`、`client_order_id`、QMT user order ID、`order_id` 和 `order_sysid` 查询/关联；
4. 将对账依据和结论写入 `audit_log`；
5. 只有明确确认无副作用或已完成账本修复后，才由人工/受控规则解锁。

绝不通过生成新的 `client_order_id` 或 `request_id` 来“试一次看看”。

### 9.3 Gateway 或 Helper 不可用

Coordinator 发现 PONG 身份不匹配、Helper 不 ready、账户数据降级、TCP 断线或可靠事件持续积压时，应停止接受新的交易副作用；可继续提供只读状态，并向策略发布统一的 `ACCOUNT_DEGRADED` / `TRADING_HALTED` 状态。恢复后先对账，再逐步解除熔断。

## 10. 实施计划

### 阶段 0：先建立治理规则

- 明确一个账户对应一个 Coordinator 实例和一个 Gateway primary；
- 禁止任何策略进程直接加载生产 `.env` 或调用 `LocalQmtApi.from_env()`；
- 为每个策略分配稳定 `strategy_id`、权限、额度和负责人；
- 规定订单、撤单、熔断、人工对账和恢复的责任边界。

### 阶段 1：实现持久化命令入口

- 创建 Coordinator 数据库迁移和上述最小表；
- 实现策略身份认证、`strategy_order_id` 幂等与 `client_order_id`/`request_id` 发行器；
- 在发送 Gateway 前完成命令账本与风险预占的原子事务；
- 初期只支持 `NEW_ASYNC`、`CANCEL_ASYNC` 和必要的只读查询。

### 阶段 2：接入唯一 Gateway 客户端

- Coordinator 独占 `qmt_local_api` 连接，并注册所有可靠事件 handler；
- 在 handler 内实现 `event_inbox + 投影 + strategy_event_outbox` 单事务落库；
- 禁止 handler 执行慢计算、网络调用或直接等待策略响应；
- 用 `delivery_id` 去重后再允许 handler 返回并 ACK。

### 阶段 3：实现策略事件 API 与风控

- 实现策略命令 API、订单查询、只读账户摘要和可靠事件订阅；
- 实现基础限额、资金/仓位/挂单预占、频率限制和熔断；
- 实现按订单归属的事件路由与策略离线 outbox；
- 保留完整审计日志和只读运维查询。

### 阶段 4：灰度切换

1. 在交易、撤单开关关闭的环境中验证握手、回调持久化、路由和重启恢复；
2. 使用模拟账户验证单策略下单、撤单、成交、重复事件和策略断线；
3. 增加第二个只读策略，再增加第二个可下单策略，验证额度冲突和优先级；
4. 小额实盘前完成告警、对账、回滚演练并获得明确审批；
5. 任何阶段都不得让旧的直连策略与 Coordinator 同时以交易身份连接 Gateway。

## 11. 验收标准

| 场景 | 通过条件 |
|---|---|
| 两个策略同时在线 | Gateway 始终只看到 Coordinator 一个 primary；两个策略都通过内部 API 获得各自事件 |
| 同一策略重复提交 | 相同 `strategy_order_id` 返回同一命令状态，不产生第二次 Gateway 副作用 |
| 跨策略 ID 冲突 | Coordinator 生成的 `client_order_id` 不冲突；手工冲突被拒绝并记录 |
| 额度冲突 | 第二条超出总资金/仓位/频率的命令在 Gateway 前被拒绝，`effect_started=false` |
| Gateway 可靠事件重复 | `event_inbox.delivery_id` 去重；策略最多收到可去重的至少一次事件 |
| 一个策略处理很慢 | 其他策略与 Gateway ACK 不被阻塞；慢策略事件留在其 outbox |
| Coordinator 重启 | 未 ACK 的 Gateway 事件重投，已 ACK 事件从本地 outbox 恢复，不重复下单 |
| 策略重启 | 只补投其未确认事件，不改变 Gateway primary |
| 未知提交 | 自动停止相关新交易、保留风险预占、完成查询对账，不生成新 ID 重发 |
| Gateway/Helper 不健康 | Coordinator 拒绝新交易副作用，恢复后先对账再开放 |

验证应先运行现有项目离线测试，再增加 Coordinator 的单元测试、崩溃恢复测试、重复投递测试和模拟账户联调。离线成功不等于大 QMT/券商实盘成功。

## 12. 回滚原则

- Coordinator 尚未向 Gateway 发送命令前，可按事务释放未执行的风险预占；
- 已进入 `GATEWAY_SUBMITTING`、`BRIDGE_QUEUED` 或未知状态的命令不可通过“删除记录”回滚，必须对账；
- 回滚到单策略模式前，先停止所有策略命令入口，确认没有未决副作用，再只启用一个被授权的 Coordinator 或直连消费者；
- 不允许旧直连策略与 Coordinator 并行成为 Gateway primary；
- Gateway、Helper、runtime、`gateway_state.sqlite3` 及其 WAL/SHM 仍作为一个状态单元保存，不单独删除或回滚。

## 13. 明确不应做的改动

1. 不要把 `runtime.primary` 改为简单列表后直接广播带 `delivery_id` 的事件；这会破坏 ACK 归属、重投和顺序语义。
2. 不要让每个策略各自使用根 `.env` 和 Gateway 令牌；这既会抢占 primary，也扩大了敏感凭据范围。
3. 不要让 Coordinator 只做内存分发；重启后无法证明哪些订单/回调已处理。
4. 不要为追求吞吐而跳过风险预占、订单映射或 `delivery_id` 落库。
5. 不要把高频行情、订单命令和可靠账户回调混在同一个无界广播队列中。
6. 不要把 QMT `passorder` 返回、`BRIDGE_QUEUED` 或 `CANCEL_SUBMITTED` 宣称为最终成交/撤单结果。

## 14. 与现有项目的衔接

本修复保留现有项目的关键边界：

- `网关/bigqmt_gateway_proxy.py` 继续提供单账户、单 primary、TCP v2、SQLite 幂等、可靠 ACK 和文件队列桥接；
- `大QMT内置python` 继续只处理 QMT 上下文工作，不增加网络服务或策略调度；
- `外置策略API/qmt_local_api` 只由 Coordinator 导入；
- 多策略能力由新增的外置 Coordinator 项目/目录承担，并以独立测试和模拟账户验收后接入。

这样既能让多个策略共享一个大 QMT 账户，又不会以牺牲现有交易安全边界换取表面上的“多客户端支持”。
