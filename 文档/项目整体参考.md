# 一台 Windows 大 QMT 内置 Python 桥接本机外置方案：项目整体参考

文档版本：v1.0  
更新日期：2026-07-18

## 1. 目标与边界

本项目允许策略、行情处理、信号计算、风控和业务数据库全部运行在大 QMT 外部的普通 Windows CPython 中。大 QMT 内置 Python 只保留必须在 QMT 上下文执行的工作：账户绑定、`passorder`、撤单、查询及订单/成交/错误回调。

项目是单 Windows、单账户、单本机消费者方案：

- Gateway 固定监听 `127.0.0.1`，不开放局域网；
- 根目录只有一份 `.env`，没有 Windows/Linux 两套配置；
- 外置策略只调用 `qmt_local_api` 或遵循 TCP v2；
- Helper 不开 socket、不创建后台线程；
- 性能、协议和可靠性基线固定在代码中，不提供可误调的部署开关；
- `.env.example` 默认账户禁用、交易禁用、撤单禁用。

它不是 QMT 的替代交易系统，也不会绕过 QMT 账户权限、交易时段、券商风控或客户端登录。外置策略仍然通过大 QMT 的正式内置接口完成交易。

## 2. 为什么仍保留 Gateway

把外置策略直接指向 Helper 文件夹看似少一层，但会丢失单 writer、原子入队、SQLite WAL 订单关联、跨重连幂等、查询 singleflight、可靠事件 ACK、身份校验和提交状态不确定保护。这些不是装饰，而是避免重复下单、错账户和丢回调的边界。

```text
┌──────────────────────────────────────────────┐
│ 普通 Windows CPython：你的外置策略           │
│ qmt_local_api / callbacks / signal / risk    │
└──────────────────────┬───────────────────────┘
                       │ 127.0.0.1:<port>
                       │ TCP v2 framed JSON
┌──────────────────────▼───────────────────────┐
│ 本机 Gateway                                  │
│ asyncio + SQLite WAL + identity + reliable ACK│
└──────────────────────┬───────────────────────┘
                       │ 本机 NTFS 原子 JSON 队列
┌──────────────────────▼───────────────────────┐
│ 大 QMT 内置 Python Helper                    │
│ run_time timers + passorder + QMT callbacks  │
└──────────────────────┬───────────────────────┘
                       │ QMT/券商交易接口
                       ▼
                    交易账户
```

同机部署去掉的是跨机器网络、Linux 进程和防火墙，不是可靠性能力。

## 4. 项目 `.venv` 与唯一 `.env`

首次部署只在项目根执行：

```powershell
.\setup_venv.ps1
```

脚本创建或复用根目录 `.venv`，验证 Windows CPython 3.12 x64，并写入 `.pth` 以离线暴露 `外置策略API`。无需激活环境，也不要对本项目执行 `pip install -e`。后续直接运行 Python 时统一使用 `& '.\.venv\Scripts\python.exe'`。

复制根模板：

```powershell
Copy-Item .env.example .env
```

解析语法是严格的 UTF-8 `KEY=value`：允许空行、整行注释和成对单/双引号；拒绝 `export`、未知键、重复键、缺项、行尾注释和非法布尔值。显式 `-EnvFile` 只负责选择哪个文件；被选中的 `.env` 内容是生产权威值，环境中残留的同名 `QMT_LOCAL_*` 不会静默覆盖它。PowerShell 启动器只读取文件中的项目 `.venv` 路径，不会把账户或令牌注入 Gateway/生成器的进程环境。

| 变量 | 用途与约束 |
|---|---|
| `QMT_LOCAL_PYTHON_EXE` | 固定为项目根相对路径 `.venv\Scripts\python.exe`；禁止系统绝对路径 |
| `QMT_LOCAL_BIND_HOST` | 必须精确为 `127.0.0.1` |
| `QMT_LOCAL_TCP_PORT` | 单账户本机端口，默认 9550，范围 1..65535 |
| `QMT_LOCAL_AUTH_TOKEN` | 外置策略首帧认证令牌，必须为随机 32 字节的 64 位十六进制；生产拒绝全零示例 |
| `QMT_LOCAL_RUNTIME_ROOT` | Helper/Gateway 文件 IPC 根目录，本机盘符绝对路径 |
| `QMT_LOCAL_LOG_DIR` | Gateway 日志目录 |
| `QMT_LOCAL_GENERATED_DIR` | 启动时生成 Gateway/QMT 配置的目录 |
| `QMT_LOCAL_HELPER_INSTALL_ROOT` | 受控部署后的 Helper 根目录 |
| `QMT_LOCAL_HELPER_OUTPUT_DIR` | Helper staging 生成目录 |
| `QMT_LOCAL_ACCOUNT_ENABLED` | 生产必须为 `true`；示例为 `false` |
| `QMT_LOCAL_ACCOUNT_NAME` | `^[A-Za-z][A-Za-z0-9_-]{0,47}$`，同时用作安全目录名与 PONG 身份 |
| `QMT_LOCAL_ACCOUNT_ID` | 真实 QMT 账号；生产拒绝 `REPLACE_*` 和全零占位值 |
| `QMT_LOCAL_ACCOUNT_TYPE` | `STOCK` 或 `CREDIT` |
| `QMT_LOCAL_HELPER_ENABLE_TRADING` | 是否允许 Helper 调用下单，默认 `false` |
| `QMT_LOCAL_HELPER_ENABLE_CANCEL_ORDER` | 是否允许 Helper 调用撤单，默认 `false` |
| `QMT_LOCAL_HELPER_STRATEGY_NAME` | 编译进 Helper 的 ASCII 策略名 |
| `QMT_LOCAL_HELPER_DEFAULT_REMARK` | 编译进 Helper 的 ASCII 默认备注 |
| `QMT_LOCAL_API_SIGNAL_JOURNAL` | 示例可靠回调 SQLite 的本机绝对路径 |
| `QMT_LOCAL_API_LOG_LEVEL` | `DEBUG/INFO/WARNING/ERROR/CRITICAL` |

用 `& '.\.venv\Scripts\python.exe' -c "import secrets; print(secrets.token_hex(32))"` 生成独立令牌并粘贴到 `.env`；不要复用账户密码、全零示例或他人项目的令牌。

`QMT_LOCAL_PYTHON_EXE` 是唯一允许的相对路径，且只能是 `.venv\Scripts\python.exe`。其余部署路径必须使用可打印 ASCII 字符，并拒绝 UNC、设备路径、相对路径、盘符根、NTFS ADS 和 Windows 非法路径字符；受管目录必须互不相同且互不嵌套，防止 Helper staging 覆写正在运行的安装目录。账户 ID、账户类型、策略名和默认备注也必须是非空可打印 ASCII。`.env` 不应提交，建议只给当前 Windows 用户和运行服务账户读取权限。

`gateway_config.json` 和 `qmt_config.json` 是从 `.env` 原子生成的实现产物，不是人工配置入口。修改它们没有支持保证，Gateway 还会再次拒绝非回环监听或被削弱的固定常量。

## 5. 固定性能与协议基线

以下值不是部署环境，不放进 `.env`：

| 位置 | 固定合同 |
|---|---|
| TCP | 协议 v2；4 字节无符号大端长度；UTF-8 JSON object；最大 10 MiB；`TCP_NODELAY`；首帧令牌/protocol/account 认证 |
| Gateway | Windows 目录通知唤醒 + 10ms 超时轮询回退；有界扫描仍是响应/事件真值；Helper 健康每 100ms 集中采样，热路径缓存最多 200ms；查询并发 1；idle 60s；SQLite WAL/NORMAL |
| Gateway I/O | 每账户文件通道 2 worker/pending 192；SQLite 通道 1 worker/pending 256；提交前固定容量检查，禁止向默认 executor 无界堆积 |
| Gateway 背压 | 每连接 dispatch 128；每账户交易副作用并发 8；命令目录硬上限采样 256、高水位 224；pending response 256；可靠响应投递 32；每轮目录扫描 1024，并轮转直查 8 个请求防止饿死 |
| API | connect/handshake 5s；recv 0.2s；心跳 5s/15s；查询 6s；pending 128；dispatcher 1024；delivery cache 8192 |
| 重连 | 0.5s、1s、2s、5s 封顶；每次重连重新完成身份校验 |
| Helper 命令 | 每 tick 最多 4；15ms 软预算；25ms 周期 |
| Helper 查询 | 每 tick 最多 1；500ms 周期；交易命令优先 |
| Helper 状态 | readiness 100ms；heartbeat 1s；reconcile 30s；maintenance 60s |
| Helper 幂等/清理 | guard 604800s；普通失败/归档文件最大 86400s，可靠 response/live event 不做 TTL 删除；每次最多删除 100、全局最多扫描 512，并轮换目录优先级；低优先级静默 1s |
| QMT 下单 | `quickTrade=2`；默认 order type 1101；QMT user order id 最长 23 |

解析 `.env` 和生成配置只发生在启动/生成阶段，热路径不读取环境变量。生成 Helper 的配置块被完全编译为常量，生成文件中禁止 `os.environ`/`os.getenv`。

## 6. 大 QMT 内置文件

`大QMT内置python/src` 是生成模板，不能直接作为生产策略文件。模板即使被误加载也强制关闭下单和撤单，配置区不读取进程环境；生成器会把配置块整块替换为根 `.env` 解析后的常量。生产产物是 ASCII、Python 3.6 语法兼容、单账户固化文件。

离线检查模板：

```powershell
.\generate_helper.ps1 -EnvFile .env.example -AllowExample
```

该命令只在系统临时目录生成并检查示例，结束后删除，不产生可部署文件。

生产步骤：

1. 运行 `.\setup_venv.ps1`，再在 `.env` 填写真实账户，生成随机 32 字节令牌填入 `QMT_LOCAL_AUTH_TOKEN`，并设 `QMT_LOCAL_ACCOUNT_ENABLED=true`；
2. 保持交易/撤单为 `false` 做首次联调；
3. 停止该账户对应的大 QMT 策略；
4. 执行：

   ```powershell
   .\generate_helper.ps1 -Deploy -ConfirmStoppedStrategy
   ```

5. 脚本先生成、再 `--check`，然后使用 staging/backup/rename 安装到：

   ```text
   <QMT_LOCAL_HELPER_INSTALL_ROOT>\<QMT_LOCAL_ACCOUNT_NAME>\
   ├─ bigqmt_file_queue_helper.py
   └─ bigqmt_loader.py
   ```

6. 在大 QMT 创建或更新专用策略，策略文件使用安装后的 `bigqmt_loader.py`，账户模型绑定到同一个 ID/type；
7. 启动策略并等待 `heartbeat.json`、`state.json`、`readiness.json`。

loader 固化 Helper 绝对路径、账户身份、build ID 和 SHA-256。Helper 被改动后 loader 会拒绝加载。修改 `.env` 中账户、路径、交易开关或备注后，必须重新生成、部署并重载大 QMT 策略；只改 `.env` 不会改变正在运行的 Helper。

## 7. Gateway 启动

大 QMT Helper ready 后运行：

```powershell
.\start_gateway.ps1
```

脚本会：

1. 严格解析根 `.env`，不从调用者进程环境继承账户或令牌；
2. 原子生成 Gateway/QMT JSON；
3. 要求当前解释器为项目 `.venv` 中的 Windows CPython 3.12 x64；
4. 检查回环端口空闲、本地路径、原子替换、SQLite WAL/NORMAL；
5. 检查三份 Helper 身份完整、一致、新鲜且 ready；
6. 启动单账户 asyncio Gateway；
7. 退出时恢复调用者原环境。

如果旧 Gateway 仍占用端口、Helper 身份不匹配或 Helper 未 ready，生产预检会 fail closed。单机模式无需管理员权限添加入站防火墙规则。

## 8. 外置策略接入

`setup_venv.ps1` 已通过 `.pth` 离线暴露稳定 API，本项目无需安装：

```powershell
.\setup_venv.ps1
```

只有外部第三方依赖需要安装，并且必须使用 `& '.\.venv\Scripts\python.exe' -m pip install <依赖名>`，避免写入系统 Python。

推荐只用稳定入口：

```python
from qmt_local_api import LocalQmtApi

api = LocalQmtApi.from_env(r"D:\bridge\.env")

def on_order(event):
    database.commit_order_event(event)  # 先持久化

api.on("ORDER_UPDATE", on_order)
api.on("TRADE_NOTIFY", on_order)

if not api.connect():
    raise RuntimeError(api.identity_guard_reason or "Gateway unavailable")

try:
    health = api.query("ACCOUNT_STATUS")
    msg_id = api.place_order_async(
        "600000.SH",
        "BUY",
        100,
        10.23,
        client_order_id="alpha-20260717-000001",
        strategy_name="alpha",
    )
finally:
    api.stop()
```

`connect()` 只有在 PONG 的 protocol/build/account ID/account name 全部匹配后才返回 ready。handler 全部成功返回后 API 才发送 `DELIVERY_ACK`；handler 抛错会主动断线且不 ACK，事件随后重投。

查询、回调、下单和撤单的可运行代码在 `示例`。下单/撤单默认 dry-run；live 模式需要精确确认字符串。

## 9. 订单幂等与状态

优先使用 `NEW_ASYNC`：

1. `ASYNC_ORDER status=SENT stage=BRIDGE_QUEUED`：Gateway 已原子排队，不等于 QMT 已接受；即时返回也可能是 `status=REJECTED stage=REJECTED` 或 `stage=SUBMIT_UNKNOWN`；
2. `ASYNC_ORDER_RESPONSE`：Helper 实际处理结果，属于可靠投递；
3. `ORDER_UPDATE`：订单状态回调；
4. `TRADE_NOTIFY`：成交回调。

每个业务订单必须有稳定且唯一的 `client_order_id`。相同 ID、相同规范意图是幂等重试；相同 ID、不同参数是 `IDEMPOTENCY_CONFLICT`。客户端通常应省略 `intent_hash`，由 Gateway 计算规范 SHA-256。

每个真实下单或撤单还必须有稳定 `request_id`。Gateway 在 SQLite 中永久绑定该 ID、动作类型与完整副作用指纹，并按 `PREPARED → DISPATCHING → ENQUEUED/UNKNOWN/TERMINAL` 推进。只有 `PREPARED` 证明尚未跨过 Helper 调度屏障；进程在 `DISPATCHING` 后崩溃时，即使最终没有调用 QMT，也只能返回 `EFFECT_STATE_UNKNOWN`，不能自动执行第二次。命令文件使用入队序号和 `request_id` SHA-256，Helper 对哈希、旧格式、processing、恢复 sibling、response 与 Guard 做有界一致性预检。

Helper v12 的 sibling 预检采用“必须证明不存在冲突”的失败关闭语义：任一目标目录无法打开、枚举中断、匹配项元数据/内容不可读，或扫描与候选预算不足，都会在 `passorder/cancel` 之前返回 `RECONCILE_REQUIRED`，并将阶段固定为 `SUBMIT_UNKNOWN`；不得把扫描失败当成“没有 sibling”后继续调用 QMT。

发生超时、断线、`SUBMIT_UNKNOWN` 或 `POST_ENQUEUE_STATE_UNCERTAIN` 时，禁止自动生成新 ID 重发。先用原 `client_order_id`、QMT user order id、订单查询和回调对账，确认没有副作用后再由业务人工决策。

异步撤单的 `ASYNC_CANCEL status=SENT cancel_status=queued` 只表示撤单请求已排队。Gateway 在文件入队前先把 pending response 写入 SQLite，因此进程重启后仍能继续等待；随后可靠投递 `ASYNC_CANCEL_RESPONSE`，表示 Helper/QMT 撤单调用结果。它仍不是券商最终撤单终态。同步撤单 `EXEC_REPORT` 中的 `cancel_status=cancel_sent, final=false` 也只表示已发送。两者最终都必须以 `ORDER_UPDATE` 的真实终态为准。

## 10. 可靠回调

`ASYNC_ORDER_RESPONSE`、`ASYNC_CANCEL_RESPONSE` 以及 Helper 文件来源的 `ORDER_UPDATE`、`TRADE_NOTIFY`、`ORDER_ERROR` 带 `delivery_id`。同名的快照差分消息可能不带该字段，属于 best-effort；只有带 `delivery_id` 的实例走下列 ACK 流程：

```text
Gateway 发送事件
  → API 唯一 reader 解帧
  → dispatcher 顺序调用该类型的全部 handler
  → handler 将事件写入业务数据库/SQLite 并提交
  → 全部成功
  → API 发送 DELIVERY_ACK(delivery_id)
  → Gateway 删除已确认的 Helper 响应/实时事件文件
```

这是至少一次投递，不是最多一次。业务库必须以 `delivery_id` 做幂等。`示例/callback_client.py` 和 `示例/common.py` 演示了 `INSERT OR IGNORE` 后返回 handler 的做法。Gateway 单次 ACK 等待窗口为 1 秒；handler 应只做快速事务落库。Helper 实时文件事件必须有稳定 `event_id`，投递失败累计三次后移入 `events/failed` 并发出 `RECONCILE_REQUIRED`；这与 `ASYNC_ORDER_RESPONSE`/`ASYNC_CANCEL_RESPONSE` 的等待和重投路径不同。

Gateway 在没有客户端时也会把 Helper response 先捕获到 SQLite；恢复连接后再可靠投递。实时事件按 `event_seq` 串行处理，较早事件失败后停止本批后续投递。handler 必须只做快速、可重复的落库；慢计算应在提交后转交独立业务队列。

同一账户只运行一个负责可靠事件的外置消费者。多个 TCP 客户端会互相替换 primary，不能当作广播订阅系统。

## 11. 安全边界

- 仅允许 IPv4 `127.0.0.1`，拒绝 `0.0.0.0`、LAN IP、IPv6 和防火墙开放；
- 首帧 PING 必须同时通过 64 位随机令牌、协议版本和账户 ID/name 校验，失败时不注册 primary，不回显令牌或账户身份；
- 令牌不替代 Windows 权限隔离；使用 NTFS ACL 将 `.env`、生成配置、runtime 和日志限定给运行账户；
- runtime、日志、SQLite、Helper 安装和生成目录必须是本机盘符、互不嵌套路径，拒绝网络盘；
- `.env`、账户 ID、生成配置、运行文件和数据库不提交；
- 控制台示例递归脱敏 account ID/name、writer token 和认证字段；
- Helper 使用 QMT 单线程 timer，不创建线程或网络服务；
- Gateway 的 writer lease 阻止同一 runtime 的第二个写者；
- QMT 账户 ID/type/name/runtime/build/protocol/command interval 任一错配都禁止交易副作用；
- 实盘前先在 Helper 和示例两侧分别显式解锁，避免单点误操作。

本项目优先防重复，不宣称网络、进程、QMT 和券商全链路 exactly-once。JSON 使用同目录临时文件加 `os.replace`，SQLite 使用 WAL + `synchronous=NORMAL`，没有为每笔文件写入增加 `fsync`；它们不承诺突然断电、磁盘缓存丢失或人工回滚状态目录后的绝对持久性。`gateway_state.sqlite3`、WAL/SHM 和 runtime 必须作为一个状态单元备份/恢复，禁止单独删除。永久 effect/订单幂等记录会增长，应监控磁盘并使用明确的停机归档方案。

## 12. 验证与验收

## 13. 推荐启动与停止顺序

启动：大 QMT 客户端登录 → 内置 Helper 策略 → `start_gateway.ps1` → 唯一外置策略/可靠回调消费者（同一 API 进程完成回调和下单）。

停止：先停止信号和下单 → 停回调消费者 → 停 Gateway → 最后停大 QMT Helper。更新 Helper 前必须先停止相关外置策略与 QMT 策略，再执行受控部署。
